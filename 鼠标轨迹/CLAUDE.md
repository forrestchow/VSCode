# K.MBox + LSTM 视觉闭环操控系统

## 项目概览

PC A 跑游戏，HDMI 环出到 PC B 采集卡；PC B 运行 CV 检测 + 决策 + LSTM 轨迹生成，通过 KMBox 向 PC A 注入键鼠操作。全程软件层面 PC A 不跑任何第三方程序。

## 核心架构

采集卡抓帧 → CV (YOLO+OCR+模板匹配) → GameState → 决策引擎 → Action(type,target) → LSTM-MDN → 轨迹点 → KMBox → PC A

### 关键：异步流水线

Frame grabber 固定 60fps，上帧 KMBox 轨迹未完成时丢弃或更新目标，避免积压延迟雪崩。

## 轨迹录制

### 原理

录制时在 PC B 侧 hook 物理鼠标事件 + 采集卡帧，时间戳对齐。LSTM 训练的是 **物理鼠标轨迹**，即人类在闭环中自然补偿了游戏变换信号后的结果。

### 工具

pynput.mouse.Listener 监听 move/click 事件，采集 `(ts, type, x, y, button)`。点击事件天然是段边界。

### 坐标系认知

```
物理鼠标轨迹（录制的）→ 游戏引擎处理（加速度/钳制/插值）→ 游戏内光标位置
```

人类大脑通过在闭环中实时补偿来应对游戏变换。LSTM 学的物理轨迹已包含补偿。**不录游戏内光标轨迹**——KMBox 发的是物理信号，经游戏变换后落点完全对不上。

但变换与帧率耦合（60fps vs 30fps 下相同物理输入产出不同光标位移），因此需 CV 兜底校正。

## LSTM 训练数据

### 分段

自动以点击事件切段。每段样本：

```python
{
    "start_pos": (x1, y1),
    "end_pos": (x2, y2),
    "action_type": "approach_npc" | "menu_select" | "confirm",
    "trajectory": [(x,y,dt), ...],
    "speed_category": "normal" | "fast"
}
```

### 模型架构

LSTM 输入层加 action_type embedding 作为条件信号。也可分类型训多个轻量模型，推理时决策引擎选模型。

### 标注

MVP 人工标注即可。规模化后可做回放工具 + 离线 CV 自动标注（帧画面判断段尾状态推断 action_type）。

### 当前数据流的关键问题（2026-06-23 确认）

**采集端** `[代码] data_collector/collector.py:174-181`：
- pynput `on_move` 由 OS 鼠标事件驱动，触发频率 = USB 回报率（125Hz→8ms，1000Hz→1ms）
- 采集到的点是**固定时间间隔**的：`(x, y, t_ms)`
- 自然结果：短距离步数少（200px→~100步），长距离步数多（1400px→~275步）

**预处理端** `[代码] training/dataset.py:95,112-128`：
- `_pad_or_truncate()` 把所有轨迹强制统一到 `SEQ_LEN=200`
- 长轨迹（T>200）：**索引均匀**重采样到 200 步（`np.linspace(0, T, 201)`），非空间弧长均匀
- 短轨迹（T<200）：末尾补零 + mask
- 时间戳 `t_ms` 被丢弃，只取 `(x, y)` 转 `(dx, dy)` `[代码] training/dataset.py:82-84`

**训练端** `[代码] training/model.py:46-59`：
- Teacher Forcing 训练，输入 condition `[sx, sy, ex, ey]`（4维），输出 `(dx, dy)` 序列（2维）
- 模型不知道每步代表多少毫秒，也不知道原始轨迹有多少步

**推理端** `[代码] demo.py:190, generate.py:49`：
- `max_steps` 硬编码为 `SEQ_LEN=200`，不论实际距离
- 短距离 50px 也生成 200 步（浪费），长距离 1400px 也生成 200 步（单步过大）

**改进方向**：
- 将 `step_count`（原始轨迹实际步数）加入 condition，扩为 5 维 `[sx, sy, ex, ey, steps]`
- 训练时分桶（bucketing）处理变长序列，避免强制截断/填充
- 推理时根据距离动态计算 steps，不再固定 200

## 执行层注意事项

- **单帧 delta 阈值**：KMBox 发送前检查每步 dx/dy，超过阈值时拉伸时间拆分步长
- **时序抖动**：KMBox 发送间隔加随机 ±10% 抖动，防 HID 时序反检测
- **CV 校正兜底**：模板匹配跟踪光标位置，落点偏差超阈值则补发小步长修正轨迹

## CV 模块

### 检测方法

| 目标 | 方法 | 原因 |
|------|------|------|
| NPC/怪物 | YOLO + 名字标签 OCR | 玩家遮挡严重时名字标签比全身更可靠 |
| 对话框/按钮 | 模板匹配 | 固定 UI 元素，毫秒级 |
| 文字选项 | OCR (PaddleOCR) | 需要知道选项内容才能决策 |
| 光标位置 | 模板匹配光标图标 | 帧差法触发检测 |

### NPC 遮挡问题

梦幻西游常见大量玩家叠加遮挡 NPC。应对：

- **名字标签 OCR 优先**：NPC 名字（黄色字体）区域小，被全遮挡概率低。OCR 识别到名字 → 偏移推测身体点击位置
- **名字 + YOLO 联合判决**：任一命中即可定位，两条路互为兜底
- **多帧聚合**：维护 30 帧窗口，站桩 NPC 短暂遮挡时取最近成功帧结果

## 游戏反脚本机制

游戏检测脚本的二维度：

| 维度 | 脚本特征 | 人类特征 |
|------|---------|---------|
| 轨迹形态 | 直线/匀速/无超调 | 弧线/变速/终点减速过头再回调 |
| HID 时序 | delta 均匀/间隔精确 | 间隔抖动/偶尔漏帧 |

游戏"移太快会卡顿/偏移"是每帧 delta 超阈值时加截断或随机偏移。
LSTM 学到的速度曲线有自然减速阶段，单帧 delta 天然小，适配速度限制。
