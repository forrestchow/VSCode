# KMBox + LSTM 视觉闭环操控系统 — 架构设计与可行性分析

## 背景

目标：构建一套**视觉驱动的闭环自动操作系统**。PC A 运行游戏，视频由 HDMI 环出到 PC B 的采集卡；PC B 运行 CV 视觉分析 + 决策逻辑 + LSTM 轨迹生成，通过 KMBox 硬件向 PC A 注入键鼠操作。全程软件层面 PC A 不跑任何第三方程序。

---

## 一、物理拓扑

```
┌──────────────────────────────────────────────────────────────────┐
│                         PC A (被控机/游戏机)                        │
│                                                                  │
│  运行游戏 ──▶ GPU渲染 ──▶ HDMI OUT ──┐                            │
│                                      │                            │
│  ◀── KMBox USB (识别为鼠标+键盘)       │                            │
└──────────────────────────────────────┼────────────────────────────┘
                                       │ HDMI 视频流
                                       ▼
┌──────────────────────────────────────┼────────────────────────────┐
│                         PC B (控制机/AI主机)                        │
│                                      │                            │
│  采集卡 HDMI IN ◀────────────────────┘                            │
│       │                                                           │
│       ▼                                                           │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌────────────┐  │
│  │ 屏幕截图  │───▶│ CV 检测   │───▶│ 决策引擎  │───▶│ LSTM 轨迹  │  │
│  │ (60fps)  │    │ (YOLO等) │    │ (策略)   │    │ 生成 (MDN) │  │
│  └──────────┘    └──────────┘    └──────────┘    └──────┬─────┘  │
│                                                        │         │
│                                                        ▼         │
│                              KMBox ◀── USB/串口 ◀── 轨迹点序列     │
│                                │                                   │
└────────────────────────────────┼───────────────────────────────────┘
                                 │ USB (被识别为键鼠HID)
                                 ▼
                    PC A 收到硬件级键鼠输入 ◀── 闭环
```

**信号流向总结**：
```
游戏画面: PC A GPU → HDMI → 采集卡 → PC B 内存 (frame buffer)
控制信号: PC B 推理 → 串口/USB → KMBox → USB HID → PC A
```

## 二、KMBox 型号选择

用户场景：**双机分离 + 视频回传**。PC B 做所有重计算，KMBox 只做执行器。

| 型号 | 双机支持 | 通信方式 | Python脚本 | 适合? |
|------|---------|---------|-----------|-------|
| **A / A+** | A+有 | HID | 无/A有宏 | ❌ HID协议能力弱，不适合逐点轨迹传输 |
| **B** | 需扩展板 | 串口 | 4MB | ⚠️ 可以但双机需额外硬件 |
| **B Pro (B+)** ★ | **原生双机** | 串口 | 4MB | ✅ **最佳选择**，USB口分离设计天然契合 |
| **Net** | 有 | 以太网 | 无 | ⚠️ 网络通信延迟更低但无Python板载 |
| **NVideo** | 有 | 以太网 | 无 | ❌ 内置采集卡重复了，用户已有HDMI采集方案 |
| **NB** | 有 | 网络+串口 | 50MB | ⚠️ 性能过冗余，除非需要8000Hz |
| **AI** | 有 | WiFi | SD卡 | ❌ 板载NPU/YOLO在本架构中用不上 |

### ★ 推荐：KMBox B Pro (B+)

理由：
- **USB口天然分离**：USB1→PC A，USB2→PC B，USB3/4→实体键鼠（需要时可人工接管）
- **串口通信**：PC B 通过 COM 口发送轨迹指令，协议成熟，Python `pyserial` 直接对接
- **双核 240MHz**：足够处理逐点轨迹缓冲和时序执行
- **1000Hz 回报率**：每点 1ms 间隔，轨迹点间隔通常 8-16ms（125Hz），绰绰有余
- **性价比**：不需要为用不上的 NPU/8K/视频采集 付溢价

## 三、全链路数据流与延迟分析

### 3.1 处理流水线（逐帧）

```
Frame N (PC A 渲染)
    │
    ├──[1] HDMI 传输延迟 ......................... <1ms (数字信号)
    ├──[2] 采集卡捕获 + DMA 到内存 ................ 5-15ms (取决于采集卡)
    ├──[3] CV 推理 (YOLO/OCR/模板匹配) ............ 5-30ms (取决于模型和GPU)
    ├──[4] 决策逻辑 (状态机/规则/RL) ............... <1ms
    ├──[5] LSTM-MDN 轨迹生成 (20-100步) ............ <5ms (GPU) / <20ms (CPU ONNX)
    ├──[6] 串口发送轨迹序列到 KMBox ................ 1-5ms (115200bps, 50点≈100字节)
    └──[7] KMBox 逐点执行鼠标移动 .................. 取决于轨迹时长 (设计目标: 100-500ms)
         │
         ▼
Frame N+1 (PC A 渲染，此时鼠标已移动到新位置)
```

### 3.2 延迟瓶颈识别

| 环节 | 延迟 | 是否瓶颈 | 优化手段 |
|------|------|---------|---------|
| HDMI 传输 | <1ms | ❌ | — |
| 采集卡捕获 | 5-15ms | ⚠️ | 选低延迟采集卡 (如 PCIe 接口的 Elgato 4K60 Pro / 圆刚 GC573) |
| CV 推理 | 5-30ms | ⚠️ | 轻量模型 (YOLOv8-nano / ONNX), GPU 推理, ROI 裁剪 |
| 决策逻辑 | <1ms | ❌ | — |
| LSTM 生成 | <5ms | ❌ | GPU 批量推理或 ONNX CPU |
| 串口发送 | 1-5ms | ❌ | — |
| KMBox 执行 | 100-500ms | ⚠️ | 这是设计变量——轨迹时长需要在"自然感"和"响应速度"间平衡 |

**关键认知**：KMBox 轨迹执行时间（100-500ms）占整个链路延迟的主导。这意味着：
- 对于**回合制/RPG/策略游戏**：500ms 延迟完全可接受
- 对于**MOBA/MMO**：200ms 左右可接受（人类反应时间 ~200-300ms）
- 对于**FPS/格斗**：需要极致优化全链路到 100ms 以内，有一定挑战

## 四、软件架构（PC B 端）

### 4.1 进程架构

```
┌────────────────────────────────────────────────┐
│                   Main Loop                     │
│           (主循环, 异步事件驱动)                  │
├────────────────────────────────────────────────┤
│                                                 │
│  Thread 1: Frame Grabber (采集卡读取)            │
│  ┌──────────────────────────────────────┐       │
│  │ 采集卡 → numpy.ndarray (HWC, BGR)      │       │
│  │ framerate: 60fps                      │       │
│  └──────────────┬───────────────────────┘       │
│                 │ frame_ready event              │
│                 ▼                                │
│  Thread 2: Vision Pipeline (CV推理)              │
│  ┌──────────────────────────────────────┐       │
│  │ YOLO 检测 → OCR 文字 → 模板匹配       │       │
│  │ 输出: GameState (entities, UI, map)   │       │
│  └──────────────┬───────────────────────┘       │
│                 │ game_state                     │
│                 ▼                                │
│  Thread 3: Decision Engine (决策)                │
│  ┌──────────────────────────────────────┐       │
│  │ 状态机/行为树/RL → Action             │       │
│  │ Action: {type: "click", target: (x,y),│       │
│  │          speed: "fast"}               │       │
│  └──────────────┬───────────────────────┘       │
│                 │ action                         │
│                 ▼                                │
│  Thread 4: Trajectory Generator (轨迹)           │
│  ┌──────────────────────────────────────┐       │
│  │ LSTM-MDN 模型推理                      │       │
│  │ 输入: (current_pos, target_pos, speed) │       │
│  │ 输出: [(x1,y1,dt1), (x2,y2,dt2), ...] │       │
│  └──────────────┬───────────────────────┘       │
│                 │ trajectory points              │
│                 ▼                                │
│  Thread 5: KMBox Executor (硬件执行)              │
│  ┌──────────────────────────────────────┐       │
│  │ 串口逐点发送, 精确时序控制             │       │
│  │ 执行完成后 fire action_done event     │       │
│  └──────────────────────────────────────┘       │
│                                                 │
└────────────────────────────────────────────────┘
```

### 4.2 关键设计决策

#### 流水线 vs 帧同步？

**建议：异步流水线模式。** 每帧不是"等上一帧执行完"，而是：
- Frame grabber 以固定 60fps 抓帧
- 如果上一帧的 KMBox 轨迹还在执行中，**丢弃**当前帧（或用新帧更新目标位置做动态轨迹修正）
- 避免积压导致延迟雪崩

#### 动态轨迹修正（高价值特性）

如果 KMBox 正在执行 A→B 的轨迹，但 CV 检测到目标移动了（B→B'），有两种策略：
1. **中断重算**：发送停止指令，从当前位置重新生成到 B' 的轨迹（可能有突兀的急转弯）
2. **在线修正**：修改剩余轨迹点的目标方向，平滑过渡（实现复杂但更自然）——适合 Phase 2+

## 五、CV 视觉分析模块设计

### 5.1 典型游戏场景的检测需求

| 检测目标 | 方法 | 模型/工具 |
|---------|------|----------|
| 游戏内物体/角色 | 目标检测 | YOLOv8-nano (ONNX) |
| UI 按钮/文字 | OCR + 模板匹配 | PaddleOCR / Tesseract + OpenCV matchTemplate |
| 地图/小地图位置 | 模板匹配 | OpenCV, 已知地图图标 |
| 血条/状态条 | 颜色阈值 + 形态学 | OpenCV |
| 屏幕变化检测 | 帧差法 | OpenCV absdiff |

### 5.2 CV 推理优化

```python
# 关键优化技巧
1. ROI 裁剪: 只推理感兴趣区域，如小地图区域、UI 栏区域
2. 跳帧推理: 不需要每帧跑 CV，可以每 2-3 帧跑一次
3. FP16/INT8 量化: YOLO ONNX 模型 int8 量化后推理 <5ms
4. 异步推理: CV 推理放独立线程，帧抓取不等待
```

## 七、核心挑战与缓解策略

### 挑战 1：全链路延迟

```
理想总延迟 = 采集(5ms) + CV(10ms) + LSTM(2ms) + 轨迹执行(150ms) = ~167ms
可接受范围: <200ms (RPG/MMO), <100ms (MOBA/动作游戏)
```

**缓解**：
- PCIe 采集卡 (≤5ms) 而非 USB (≥30ms)
- YOLO 轻量模型 + FP16 + ROI 裁剪
- 短距离移动用短轨迹 (<50pt)，长距离才用全轨迹

### 挑战 2：CV 检测的鲁棒性

同一物体在不同光照/特效/皮肤下可能检测失败。

**缓解**：
- 多模型级联：先跑快速的模板匹配，置信度低时才跑 YOLO
- 时序平滑：连续 N 帧检测结果取平均/中值，过滤单帧误检
- 采集 1000+ 张游戏截图做数据增强后微调 YOLO

### 挑战 3：轨迹的"上下文感知"

LSTM 只知起点终点，不知屏幕上的障碍物（UI 面板、地形）。

**缓解**：
- 决策层判断是否需要"绕行"，如果需要，生成中间途经点 (waypoint)，分段生成轨迹
- 轨迹的后处理过滤器：检测生成轨迹是否穿过"禁区"，如有则拒绝+重新生成

### 挑战 4：反作弊检测

这个架构比单机软件方案隐蔽得多，但仍有理论风险：

| 检测向量 | 风险等级 | 缓解 |
|---------|---------|------|
| USB HID 描述符指纹 | 低 | KMBox 使用通用 HID 描述符 |
| 鼠标轨迹统计特征 | **高** | LSTM-MDN 多样性 + 对抗训练 |
| 行为一致性 (总是相同延迟) | 中 | 轨迹时序加随机抖动 |
| 视频采集卡检测 | 极低 | 采集卡是纯接收设备，无信号输出 |

## 十、为什么不选择其他方案

| 替代方案 | 劣势 |
|---------|------|
| 单机软件 (SendInput/autogui) | 被反作弊检测的风险极高；注入到游戏进程空间会触发完整性检查 |
| DMA 硬件 | 需要定制固件 + FPGA 开发，成本 ¥2000+，技术门槛极高 |
| Arduino Leonardo 模拟 | 仅支持简单宏，无 Python 扩展能力，不能逐点轨迹 |
| 直接远程桌面/VNC | 画面延迟大，键鼠输入特征明显不同 |
