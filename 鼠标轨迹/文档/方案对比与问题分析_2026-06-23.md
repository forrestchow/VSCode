## 三、蠕动问题

### 3.1 原因 `[代码] + [推理]`

```python
# demo.py:190 — 永远生成 200 步
deltas = self.model(condition, max_steps=SEQ_LEN).squeeze(0).numpy()
```

- 模型在训练时学到「有效 delta 只有约 89 步（均值），后面是零」 `[代码] dataset.py:129-136`
- 但推理时自回归循环固定 200 次 `[代码] model.py:105-110`
- 前 89 步走完终点，后 111 步无事可做
- 自回归误差自反馈 → 终点附近微微抖动 → **蠕动**

### 3.2 当前方案能否解决

**否。** 截断 `max_steps`（如设为 62）会让模型在到达终点前就被截断——模型不知道该「加速」走完，只走了约 70% 的路程。

### 3.3 方案 A/B 如何解决

```
模型学到:  condition.steps=62  → 62步走完全程
             condition.steps=89  → 89步走完全程
             condition.steps=120 → 120步走完全程

推理时:    steps 由距离统计得出
             max_steps = steps
             循环恰好 steps 次 → 无多余生成空间 → 无蠕动
```

`[推理]` 根除原理：**没有多出来的步数，就没有蠕动的物理空间。**

### 3.4 不改模型的临时止血方案

```python
# 模型照常生成 200 步，通过位移累计找到有效切割点
cumsum = np.cumsum(deltas, axis=0)
total_disp = np.sqrt(cumsum[:, 0]**2 + cumsum[:, 1]**2)
cutoff = np.searchsorted(total_disp, total_disp[-1] * 0.98) + 1
deltas = deltas[:cutoff]  # 裁掉尾部零位移区
```

`[推理]` 不解决根本问题，但可以止血。依赖模型确实在末尾输出趋零 delta。

## 四、频率问题

### 4.1 问题的本质 `[推理]`

模型输出的是纯几何量 `(dx, dy)`，没有时间维度。三个环节有三个不同的频率：

```
环节        频率              单位帧间隔      每步含义
────        ────              ────────       ──────
采集        ~64 Hz            15.6 ms        物理鼠标事件间隔
模型        无频率概念         不存在         纯数字序列 (dx,dy)
KMBox 发送   目标 1000 Hz      1 ms           HID 报告间隔
```

### 4.2 方案 A/B 能否解决

**不能。** 方案 A/B 解决的是「生成多少步」（空间颗粒度），不解决「每步代表多少毫秒」（时间标度）。

```
500px 距离，方案 A 推理 steps=89:

  若 64Hz 发送 (15.6ms/步) → 89×15.6=1387ms → 360 px/s  ← 匹配采集者
  若 125Hz 发送 (8ms/步)   → 89×8=712ms     → 702 px/s  ← 快了 2 倍
  若 1000Hz 发送 (1ms/步)  → 89×1=89ms      → 5618 px/s ← 瞬移

同样的 89 步模型输出，发送频率不同，速度天差地别。
```

### 4.3 完整的频率解决方案 `[推理]`

**三层架构，缺一不可：**

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  模型层 (方案 A)                                         │
│  ─────────────                                          │
│  condition = [sx, sy, ex, ey, steps]                    │
│  模型输出 steps 个控制点 delta                           │
│  来自训练数据统计: steps = 43.4 + 0.092 × distance      │
│  作用: 确定「走多少步」，形状自然                         │
│                                                         │
│            ↓                                            │
│                                                         │
│  CR 插值层 (密度桥接)                                     │
│  ────────────────────                                    │
│  目标: 每发送帧位移 ≤ 0.36px (肉眼不可见 @1000Hz)         │
│  cr_factor = ceil(每步delta / 0.36)                      │
│  steps 个控制点 → steps × cr_factor 个发送点             │
│  作用: 填满物理密度，接上 KMBox 的高频时钟                │
│                                                         │
│            ↓                                            │
│                                                         │
│  KMBox 发送层 (固定时钟)                                   │
│  ─────────────────────                                   │
│  固定 1000 Hz 发送，每 1ms 一帧                           │
│  总耗时 = 发送点数 × 1ms                                  │
│  速度 = 距离 / 总耗时 ≈ 355 px/s (采集者手速)             │
│  作用: 提供均匀时间基准，自然速度                          │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 4.4 为什么模型不能直接输出高频步数

`[推理]`

- 采集频率 ~64Hz 决定了训练数据最多 132 步
- KMBox 1000Hz 需要 ~1400 步（差距 10 倍）
- 让模型生成 1400 步 = LSTM 循环 1400 次 = 慢 + 误差大
- 模型步数 = 控制点（管形状），KMBox 发送数 = 物理帧（管密度），两者天生不同

## 六、核心结论

1. **蠕动**：方案 A/B 根除。不改模型用裁切法止血。 `[推理]`
2. **频率**：任何方案都需要三层架构（模型 + CR + 固定 Hz 发送），单靠改模型无法解决。 `[推理]`
3. **steps 预测**：`steps = 43.4 + 0.092 × distance_px`，来自训练数据线性拟合（r=0.68）。 `[数据]`
4. **当前数据量太小**：33 条、单采集者、~64Hz，只能做概念验证。 `[数据]`
