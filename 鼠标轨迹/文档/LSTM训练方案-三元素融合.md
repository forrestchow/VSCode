# LSTM 训练方案：三元素融合

## 问题描述

鼠标轨迹训练的输入元素有三个：

| 元素 | 来源 | 训练时可得 | 推理时可得 |
|------|------|-----------|-----------|
| **物理轨迹** | pynput 录制 | ✅ | KMBox 发出的结果是这个 |
| **画面轨迹** | 采集卡帧 + 模板匹配跟踪光标 | ✅ | **否**（推理时要生成的目标） |
| **画面场景** | 采集卡帧 + CV/OCR/YOLO 推断 action_type | ✅ | ✅（CV 实时检测） |

传统 LSTM 方案只用到物理轨迹 + 画面场景（作为 action_embedding），**画面轨迹**从未在传统方案中出现过。问题在于：

> 画面轨迹既不能作为模型输入（推理时不可得），又不能简单地丢弃（它包含了游戏变换的关键信息），那它怎么用？

---

## 方案 A：辅助 Loss（多任务学习）

### 架构

```
输入: [screen_start_x, screen_start_y, screen_target_x, screen_target_y, action_embedding]
           │
           ▼
    ┌─────────────┐
    │  LSTM 编码器  │  ← 共享参数
    └──────┬──────┘
           │
     ┌─────┴─────┐
     ▼           ▼
  物理轨迹头    画面轨迹头
  (主输出头)    (辅助输出头)
     │           │
     ▼           ▼
  物理Loss_p   画面Loss_s
     │           │
     └─────┬─────┘
           ▼
   总 Loss = Loss_p + λ * Loss_s
          ↑ λ 控制画面轨迹的监督强度
```

### 训练数据要求

一条样本需要同时包含：

```python
{
    "screen_start": (820, 210),
    "screen_target": (500, 300),
    "action_type": "approach_npc",
    "physical_trajectory": [
        (0, 820, 210), (8, 812, 208), ..., (1234, 500, 300)
    ],
    "screen_cursor_trajectory": [    # ← 必须逐点对齐
        (0, 820, 210), (8, 818, 208), ..., (1234, 500, 300)
    ],
}
```

**画面轨迹需要用模板匹配逐帧跟踪光标来获取，与物理轨迹在时间戳上逐点对齐。**

### 推理时

```
CV → screen_start, screen_target → 决策引擎 → action_type
                                    ↓
                               LSTM（只用物理头）
                                    ↓
                               physical_trajectory → KMBox → PC A
```

画面轨迹头在推理时被完全丢弃。画面轨迹只参与训练时的 Loss 计算。

### 优缺点

| | 内容 |
|---|---|
| ✅ | 不改 LSTM 本质结构，架构改动小 |
| ✅ | 画面轨迹信息通过梯度反向传播被编码器吸收，隐式学到了"物理→画面"的变换 |
| ✅ | 推理延迟不变（仍是一次 ONNX 前向） |
| ❌ | 画面轨迹必须逐点对齐，数据工程工作量大 |
| ❌ | λ 需要调参，设太大模型可能偏向画面轨迹而牺牲物理轨迹质量 |
| ❌ | 画面轨迹对主任务的提升难以量化评估 |

## 方案 B：两阶段串行

### 架构

```
阶段 1（路径生成器）:
输入: [screen_start_x, screen_start_y, screen_target_x, screen_target_y, action_embedding]
           │
           ▼
    ┌──────────────┐
    │  LSTM_A 编码器 │
    └──────┬───────┘
           ▼
    screen_cursor_trajectory      ← 中间产物：画面光标轨迹

阶段 2（物理映射器）:
输入: screen_cursor_trajectory    ← 消费阶段 1 的输出
           │
           ▼
    ┌──────────────┐
    │  LSTM_B 编码器 │
    └──────┬───────┘
           ▼
    physical_trajectory           ← 最终产物：KMBox 要发送的
```

两个模型独立训练：

- **LSTM_A 训练**：`(screen_start, screen_target, action_type) → 画面轨迹`。目标函数是画面的光标路径拟合。本质上在学"人在画面上会怎么移动光标"。
- **LSTM_B 训练**：`(画面轨迹序列) → 物理轨迹序列`。目标函数是物理信号的拟合。本质上在学"游戏的逆变换"——给定的画面光标路径，需要什么物理输入才能触发它。

### 训练数据要求

LSTM_A 所需：

```python
{
    "screen_start": (820, 210),
    "screen_target": (500, 300),
    "action_type": "approach_npc",
    "screen_cursor_trajectory": [(0, 820, 210), ..., (1234, 500, 300)],
}
```

LSTM_B 所需：

```python
{
    "screen_cursor_trajectory": [(0, 820, 210), ..., (1234, 500, 300)],
    "physical_trajectory": [(0, 820, 210), ..., (1234, 500, 300)],
}
```

画面轨迹是 LSTM_A 的输出和 LSTM_B 的输入，是两张表之间的**桥梁**。

### 推理时

```
CV → screen_start / screen_target → 决策引擎 → action_type
                                    ↓
                               LSTM_A
                                    ↓
                            screen_trajectory    ← 显式中间结果
                                    ↓
                               LSTM_B
                                    ↓
                            physical_trajectory → KMBox → PC A
```

两阶段串联，两个推理调用。

### 优缺点

| | 内容 |
|---|---|
| ✅ | 建模更显式——画面轨迹是物理可见的中间结果，可调试、可验证 |
| ✅ | 如果游戏更新了鼠标处理逻辑，只需重训 LSTM_B（物理映射器），LSTM_A 可复用 |
| ✅ | 画面轨迹本身也有用——可用于 CV 校正器判断"轨迹是否偏了" |
| ❌ | 两阶段串联，推理延迟翻倍 |
| ❌ | 误差可累积：LSTM_A 的输出偏差 → LSTM_B 的输入偏差 → 最终轨迹双重偏差 |
| ❌ | 需要维护两个模型，工程复杂度高 |
| ❌ | LSTM_B 本质上在学查表——游戏变换的逆向映射可能不是一个平滑的序列函数 |

---

## 两方案对比总表

| 维度 | 方案 A：辅助 Loss | 方案 B：两阶段串行 |
|------|-----------------|-----------------|
| 模型个数 | 1 个 | 2 个 |
| 推理延迟 | 1× ONNX | 2× ONNX |
| 工程改动量 | 小（加一个输出头 + Loss） | 大（两套训练管线、两次推理） |
| 画面轨迹作用 | 辅助正则化信号 | 显式中间表示 |
| 位置误差（游戏光标 vs 模拟光标） | LSTM 隐式容错 | LSTM_B 显式学映射 |
| 训练数据对齐要求 | 严格逐点对齐 | 两个阶段分别对齐 |
| 可调试性 | 差（画面迹被 Loss 吸收） | 好（中间结果可看） |
| 游戏更新后 | 必须全模型重训 | 可能只需重训 LSTM_B |

---

## 推荐建议

**MVP（尽快验证可行性）：用方案 A。**

理由：
- 一个 LSTM 就够了，改动最小
- 延迟最低（推理一次）
- 快速验证"跨坐标系隐式学习"是否有效

**如果发现游戏变换不稳定（掉帧/负载波动导致物理→画面映射不一致）：切方案 B。**

理由：
- 两阶段能独立调试 LSTM_B 的短板
- 画面轨迹作为中间层可以人工检查和修正后喂给 LSTM_B，而不是放大误差
