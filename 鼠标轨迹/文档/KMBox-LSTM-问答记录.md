# KMBox + LSTM 鼠标轨迹 — 问答记录

> 2026-06-17 对话整理

---

## Q1: KMBox 控制鼠标的原理和 LSTM 模型生成轨迹是否一致？

**答：完全一致。** KMBox 本质上是一个"坐标点播放器"——接收 `(dx, dy, 间隔时间)` 序列，按时序转换成 USB HID 鼠标报告发出。

```
LSTM 模型输出                    KMBox 执行
─────────────────────────────────────────────
(dx₁, dy₁)  ──────────────────→  HID Report #1
(dx₂, dy₂)  ──────────────────→  HID Report #2
   ...
(dx₂₀₀, dy₂₀₀) ───────────────→  HID Report #200
```

USB 鼠标协议本身就是**相对位移**，不存在绝对坐标。KMBox 把软件告诉它的 delta 值用硬件级别的 HID 包发出去，与模型输出的 delta 序列天然一致。

这也是项目录制"物理轨迹"而非"游戏内光标轨迹"的根本原因——人类手部移动产生的原始 delta 序列被 LSTM 学习，推理时生成的 delta 由 KMBox 原样发出，人类在闭环中实时补偿游戏变换的模式已被隐式学到。

### 额外工程考量
- **单帧 delta 阈值**：KMBox 一次发送的 dx/dy 若超出人类单帧移动能力，需在时间上拆分
- **时序抖动 ±10%**：防止 HID 报告间隔完全均匀（明显脚本特征）

## Q2: 能否先用 KMBox 连接本机进行开发测试？

**答：可以，且是最合理的开发测试方式。**

### 连接拓扑

```
你的 PC
  ├── USB 口 ① ← 物理鼠标/键盘（正常使用）
  ├── USB 口 ② ← KMBox（系统识别为"第二套键鼠"）
  └── Python 程序 ──串口/网口──→ KMBox
```

KMBox 插到本机 USB 后，Windows 将其识别为普通 USB 鼠标+键盘，不区分是真实硬件还是 KMBox。

### 本机测试 vs 双机方案

| | 本机测试 | 双机方案 |
|---|---|---|
| 硬件成本 | 1 台 PC + KMBox | 2 台 PC + KMBox + 采集卡 |
| 调试效率 | 直接 print/log，秒级迭代 | 采集卡画面回传，链路长 |
| 适用阶段 | **开发 & 验证模型输出** | 部署 & 反检测验证 |

### 注意点
- KMBox 控制的光标和物理鼠标**共享同一个光标**，不是独立的
- 开发时手不要碰物理鼠标，否则会和 KMBox 抢光标
- LSTM 输出本身就是 delta（相对位移），与 KMBox HID 协议天然一致，无需改动

### 建议上手路径
1. 买 KMBox B Pro（B+），获取 SDK（通常是串口通信 DLL 或 Python 示例）
2. 插本机 USB，跑卖家 demo 确认能移动光标
3. 将 `demo.py` 生成的 delta 序列通过 KMBox SDK 发送，替换画布动画
4. 打开画图/记事本观察真实光标是否走出 LSTM 轨迹

## Q3: YOLO 返回的游戏元素坐标，转换后如何控制鼠标？

**答：整个链路如下：**

```
采集卡画面 (1920×1080)
      │
      ▼
YOLO 检测 ──→ NPC 名字标签在像素 (1200, 400)
OCR 识别   ──→ "店小二" → 决策引擎判定"应该点击"
模板匹配   ──→ 当前光标在像素 (300, 500)
      │
      ▼
LSTM 输入: [300/1920, 500/1080, 1200/1920, 400/1080]
      │  条件向量 [sx, sy, ex, ey] 归一化
      ▼
LSTM 输出: [(dx₁,dy₁), ..., (dx₂₀₀,dy₂₀₀)]
      │  物理鼠标相对位移序列
      ▼
KMBox 逐帧发送 HID Report ──→ PC A 上光标移动
```

### 关键陷阱：画面坐标 ≠ 物理距离

YOLO 返回的是**游戏画面上的像素位置**，KMBox 发出的 delta 是**物理鼠标信号**。二者经游戏引擎变换后不线性对应。

但 LSTM 训练数据本身就是物理轨迹，模型学到的映射已**隐式包含游戏变换的补偿**：

```
采集阶段：手移动鼠标 500px（物理）→ 画面光标移动 300px
LSTM 学到：画面偏离 300px 时，应发约 500px 的物理 delta

推理阶段：YOLO 说目标在画面偏移 (600, 300) 处
→ LSTM 自动输出物理 delta ≈ (1000, 500)  ← 自动放大了比例
```

**不需要手动换算坐标系。**

### 桌面模式 vs 游戏模式

当前已实现的**桌面模式**不存在这个问题：

```
桌面模式（已实现）：
  画布坐标 = 物理鼠标映射，1:1 线性
  pynput 录制物理 delta → LSTM 学习 → 推理输出 delta → 累加落点
  开环精度已经很高，无需 CV 反馈
```

**游戏模式**才会出现：

```
游戏模式（未来）：
  KMBox 发物理 delta 500px → 游戏引擎处理（加速度/钳制/插值）
  → 游戏内光标只移动了 300px

  帧率波动（30fps vs 60fps）下相同物理输入产出不同光标位移
  游戏内部有随机抖动/平滑插值
  → 需要 CV 反馈来兜底
```

### CV 反馈修正系统

当游戏模式出现坐标偏移时，需要引入反馈修正——但关键是**怎么做**。全闭环不行，间歇兜底才是正解。

#### 三种方案对比

**方案 A：纯开环（无反馈）**
```
LSTM 生成 → KMBox 播放 → 结束
```
问题：帧率波动、游戏随机抖动都会累积偏差，落点不可控。

**方案 B：逐帧闭环（每帧反馈）**
```
CV 查位置 → 算偏差 → 发一个 delta → CV 再查 → 再发 → ...
  16ms    32ms    48ms  ...
```
问题：CV 检测 + 采集卡延迟 > 16ms，60fps 下积压延迟雪崩，根本跟不上。

**方案 C：开环 + 纠偏（项目采用）**
```
LSTM 生成整条轨迹 → KMBox 播放完 → CV 查一次落点
  ├── 偏差 < 阈值 → 结束
  └── 偏差 > 阈值 → 补发 1-2 个小步长修正
```

#### 为什么方案 C 是最优解

LSTM 模型已经学了补偿模式——输出的物理 delta 经过游戏变换后**大概率**落在目标附近。偏差只是"误差残余"，通常几十像素以内，一轮微调就够。

```
时间线：

  0ms  ─────────────────── 800ms ── 820ms ── 850ms
  │                         │       │        │
  │  LSTM 200点轨迹播放      │       │        │
  └────────────────────────→│       │        │
                             │       │        │
                           CV 检测  偏差 15px │
                             │       │ 补发   │
                             └───────┴───────→│ 完成
```

核心思路：**LSTM 做 95% 的活，CV 只做最后 5% 的纠偏。** 不是"要不要反馈"的问题，而是反馈只用一两次，不是每帧都用。这在桌面模式下不需要（开环精度已经很高），等接入 KMBox 对着游戏跑的时候再上。
